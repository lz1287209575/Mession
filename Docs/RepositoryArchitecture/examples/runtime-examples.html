示例 3：World Runtime Dispatch 入口

这里能看到 World 对外暴露的不是 Workflow 类，而是 Runtime Dispatch 契约：先校验和准备，再把业务体排进 player command runtime。

Dispatch Runtime CommandSource/Servers/World/Player/PlayerSwitchScene.cpp
MFuture<TResult<FPlayerSwitchSceneResponse, FAppError>> MPlayerService::PlayerSwitchScene(
    const FPlayerSwitchSceneRequest& Request)
{
    return DispatchRuntimeCommand<FPlayerSwitchSceneResponse>(
        Request,
        "PlayerSwitchScene",
        { EDependency::Scene, EDependency::Router },
        &MPlayerService::DoPlayerSwitchScene);
}

示例 4：Fiber 风格异步业务函数体

这段代码最能说明你想要的方向：写业务的人看到的是顺序代码，异步复杂度被 Runtime 吃掉，而不是再回到 Callback Hell。

Scene Call with MAwaitOkSource/Servers/World/Player/PlayerCombat.cpp
return MServerCallAsyncSupport::StartFiber<FWorldCreateCombatAvatarResponse>(
    WorldServer->GetTaskRunner(),
    [this, SceneRequest = std::move(SceneRequest)]() mutable
    {
        const FSceneSpawnCombatAvatarResponse Value =
            MAwaitOk(WorldServer->GetScene()->SpawnCombatAvatar(SceneRequest));

        FWorldCreateCombatAvatarResponse Response;
        Response.PlayerId = Value.PlayerId;
        Response.SceneId = Value.SceneId;
        Response.CombatEntityId = Value.CombatEntityId;
        return TResult<FWorldCreateCombatAvatarResponse, FAppError>::Ok(std::move(Response));
    });
从写法上看，这更像 async / await 的降阶版体验，只是底层依赖的是你们自己在 C++17 上做的 fiber runtime。
如果后续新业务也能保持这种风格，Workflow 类就没有必要再长回来。
真正应该继续投资的是 Runtime 能力、请求校验和 shared async glue，而不是再发明新的流程对象层。
