基础层总览

这一层决定的是系统“怎么跑”，而不是具体业务“做什么”。

Common

  • Runtime/Concurrency：线程池、task queue、fiber scheduler、await。
  • Net：连接、server base、RPC transport。
  • Runtime/Object / Reflect / Replication：对象、反射、复制。
  • Serialization：消息编解码。

Protocol

  • 按 Auth / Gateway / World / Scene / Combat / Router / Mgo 切分消息。
  • 它是跨模块边界的契约层，不应被业务实现反向污染。
  • 后续要继续防止 Messages/World 过胖。

Servers/App

  • ServerCallAsyncSupport / ClientCallAsyncSupport：server/client 调用边界。
  • FiberCallSupport：统一 fiber bootstrap。
  • ResultFutureSupport：统一 future/result combinator。
  • ObjectCall*：对象级路由调用。

Tooling / Build

  • MHeaderTool 生成反射与 dispatch glue。
  • CMakeLists.txt 把仓库拆成 core/common/netdriver 和各 server。
  • 当前难点不是能力不够，而是生成链理解门槛较高。

基础层评价

这几层现在整体是健康的，但也最适合继续做“公共化”和“去重复”。

优点

  • 已经形成了独立于业务 server 的通用底座。
  • 并发与异步能力正在逐步从 World 抽回 Common/App。

风险

  • 若命名继续带 World 语义，会限制公共层进一步抽象。
  • 生成链和反射链调试成本偏高。

建议

  • 继续公共化命名和 helper。
  • 补一页生成链说明，降低理解门槛。

这里不再只讲抽象概念，而是直接放一些来自仓库的真实接口形态，帮助你确认 Runtime / App 层到底承担了什么责任。

Request Validation HelperSource/Servers/App/ServerCallRequestValidation.h
template<typename TRequest>
TOptional<FAppError> ValidateRequest(const TRequest& Request)
{
    return TRequestValidator<std::decay_t<TRequest>>::Validate(Request);
}

inline TOptional<FAppError> RequireNonZero(uint64 Value, const char* Code, const char* Message = "")
{
    if (Value == 0)
    {
        return FAppError::Make(Code ? Code : "request_field_required", Message ? Message : "");
    }

    return std::nullopt;
}
Server Call Chain HelperSource/Servers/App/ServerCallAsyncSupport.h
template<typename TInputResponse, typename TBinder, typename TNextFuture, typename TOutputResponse>
TAppFuture<TOutputResponse> Chain(TAppFuture<TInputResponse> Future, TBinder&& Binder)
{
    return MAppResultFutureSupport::Chain(
        std::move(Future),
        std::forward<TBinder>(Binder),
        [](const char* Message)
        {
            return MDetail::MakeExceptionFuture<TOutputResponse>(Message ? Message : "invalid_future");
        },
        [](const char* Message)
        {
            return MDetail::MakeAsyncExceptionError(Message);
        });
}

源码定位提示

如果你想顺着这些例子继续往下看实现细节，推荐的入口顺序是 FiberAwait.h → ServerCallRequestValidation.h → ServerCallAsyncSupport.h，然后再跳到 runtime/world-runtime.html 看它们如何在 World 里组合起来。

Common 负责 await 能力与 fiber 运行时。
Servers/App 负责请求校验和 async glue 的统一抽象。
World 则把这些底座能力装配成玩家命令执行模型。
