World Runtime

这套设计的核心，不是再发明一个新 Workflow，而是把异步和并发复杂度往 Runtime 层压

你想要的不是“另一个流程对象”，而是让业务代码看起来像顺序代码；目前这套 C++17 Runtime / fiber / strand / epoch 的组合，本质上就是在做这件事。

设计目标

这页先把 World Runtime 真正解决什么问题讲清楚。

不要 Workflow

避免业务逻辑再拆成一堆 OnStep / 状态跳转 / 回调链。

要顺序代码

让 DoXxx(...) 看起来像同步流程，业务意图能从上往下读。

要运行时兜底

并发互斥、过期命令剔除、异步挂起恢复，都由 Runtime 统一负责。

对外接口应该长什么样

World 入口暴露的不是状态机接口，而是“校验 → 准备 → 排队执行”的 Runtime Dispatch 契约。

Fiber 与调度器负责什么

命令真正跑起来之后，挂起 / 恢复由 fiber scheduler 接管，这样业务函数体才能写成顺序形式。

当前已知边界

这套 Runtime 已经可用，但当前仓库里仍有几条需要明确写出来的边界，避免误把“当前可跑”理解成“语义已经完全收口”。

当前支持的语义

  • PlayerCommandRuntime 负责按玩家串行化执行。
  • 多参与者命令会先通过 strand barrier，再统一启动 fiber。
  • MAwait / MAwaitOk 支持在当前 player command fiber 中挂起和恢复。
  • epoch 变化会把旧命令视为 stale，避免过期命令继续提交。

当前仍未完全收口的点

  • PlayerCommandRuntime 现在同时承担串行化、多人 barrier、fiber 生命周期包装，职责还偏重。
  • FiberScheduler 在不同平台上实现能力仍有差异。
  • 运行时专项验证刚开始分层，当前更多还是从总回归里切出的 suite。

World 调用边界现状

近期这轮整理不只是收 Runtime，也在收 WorldClient → PlayerService → ObjectCall 这条调用边界，避免复杂度在几个入口之间来回漂移。

WorldClient*

  • 只负责 client-call 适配和回包。
  • 普通 Player 请求通过 WorldClientCommon::StartAsyncClientResponse 统一进入运行时。
  • 不再鼓励在 Client_* 里手写重复的 deferred glue。

PlayerService

  • 暴露 Runtime Dispatch 契约。
  • 统一承接校验、依赖检查、命令排队。
  • 真实业务步骤继续下沉到 DoXxx(...)。

ObjectCall*

  • 负责对象级本地调用和远端代理调用。
  • 当前已拆出 target server resolve、本地 target resolve、payload parse helper。
  • ObjectCallRouter 已与主调用层共用本地 resolve 语义。
当前要持续防的不是“缺少抽象”，而是同一条调用链在多个 owner 里各自再写一遍 transport glue。
如果后续新增命令，优先判断它属于 WorldClient 适配、PlayerService 调度，还是 ObjectCall 代理，不要混写。

当前风险与验证入口

后续改 Runtime 相关代码时，不应该只看代码风格是否统一，更重要的是同步跑对应回归。

已知最高风险点之一：Windows fiber backend 目前仍是占位实现，不应把它当作完整 stackful fiber 支持。
已知最高风险点之二：PlayerCommandRuntime 里串行化、barrier、epoch、异常传播的职责还缠在一起，后续拆分要谨慎。
已知最高风险点之三：ObjectCall 虽然已开始拆 helper，但本地 dispatch bridge 和远端 proxy 仍在同一层，后续继续收时要保持错误模型一致。
当前推荐先跑：python3 Scripts/validate.py --build-dir Build --no-build --suite runtime_dispatch，再跑全量 all。

为什么这套 Runtime 不会退化成另一种 Workflow

关键不在于有没有多个异步步骤，而在于这些步骤是不是仍然由统一 Runtime 驱动，而不是回到业务层自己管理状态推进。

多步异步本身不是问题，问题是让业务自己保存“下一步该做什么”。
现在这套设计里，业务只写 MAwaitOk 之前和之后的直线逻辑，挂起与恢复由 fiber runtime 负责。
所以复杂度沉到了 Runtime，而不是重新沉到 Workflow class 或 callback graph。
Start Barrier And Stale CheckSource/Servers/World/Player/PlayerCommandRuntime.cpp
TSharedPtr<MFiberExecution> Execution = MFiberScheduler::Get().CreateExecution(
    {
        Command,
        std::move(Context),
        [Command, Start = std::move(Start)]() mutable
        {
            for (const SPlayerCommandEpoch& Participant : Command->Participants)
            {
                if (Command->Runtime->SnapshotEpoch(Participant.PlayerId) != Participant.Epoch)
                {
                    throw std::runtime_error("player command became stale before execution");
                }
            }

            Start(Command);
        },
        [Command]()
        {
            for (const TSharedPtr<MPlayerStrand>& Strand : Command->Strands)
            {
                if (Strand)
                {
                    Strand->OnCommandCompleted();
                }
            }
        },
        [Command, FailureCallback](std::exception_ptr Exception) mutable
        {
            if (FailureCallback)
            {
                FailureCallback(FAppError::Make("player_command_exception", "unknown"));
            }
        },
    });

扩展一个新业务命令时的标准写法

如果后续再接新业务，建议严格保持这一套接口形态，不要给新命令发明特例。

Recommended Extension PatternWorld Player Service Pattern
MFuture<TResult<FYourResponse, FAppError>> MPlayerService::YourCommand(const FYourRequest& Request)
{
    return DispatchRuntimeCommand<FYourResponse>(
        Request,
        "YourCommand",
        {
            EDependency::Scene,
            EDependency::Router,
        },
        &MPlayerService::DoYourCommand);
}

TResult<FYourResponse, FAppError> MPlayerService::DoYourCommand(FYourRequest Request)
{
    const FSceneResponse SceneResponse = MAwaitOk(WorldServer->GetScene()->YourRpc(...));
    const TResult<FPlayerUpdateRouteResponse, FAppError> RouteResult =
        ApplySceneRouteForPlayer(Request.PlayerId, SceneResponse.SceneId);
    return RouteResult.IsOk()
        ? TResult<FYourResponse, FAppError>::Ok(FYourResponse{})
        : MakeErrorResult<FYourResponse>(RouteResult.GetError());
}

入口层职责

  • 做 ValidateRequest、依赖可用性检查。
  • 把命令统一交给 DispatchRuntimeCommand / DispatchRuntimeCommandMany。
  • 不在入口层手工堆 callback graph。

业务体职责

  • 只表达真正的领域步骤。
  • 异步点用 MAwait / MAwaitOk。
  • 错误通过 TResult<..., FAppError> 返回，不另外发明流程状态。

我对这条路线的判断

如果目标是在 C++17 下尽量逼近 async / await 的开发体验，而又不想退回 Workflow 类，这条路目前是合理的。

为什么值得继续

  • 它直接改善业务可读性。
  • 它把复杂度放在少数 Runtime 文件里集中治理。
  • 它天然鼓励 Servers/App 继续抽公共 helper。

真正要持续防守的点

  • 不要让 MPlayerService 无限制长胖。
  • 不要把 Runtime 特殊逻辑散落回每个业务函数。
  • 不要用新的流程 owner 包一层把旧问题带回来。
