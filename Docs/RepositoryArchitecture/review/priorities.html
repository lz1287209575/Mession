优先级总表

这个表更接近架构 review 清单，而不是功能待办表。

模块优先级当前主要问题原因
WorldP0中心复杂度持续累积，MPlayerService 仍有继续膨胀风险。所有玩家主链路都会汇聚到这里。
Servers/AppP0边界 glue 仍是全仓库复用收益最大的地方。每减少一份重复代码，所有 server 都受益。
CommonP1runtime 基础设施仍残留少量偏业务命名。会限制下一轮公共抽象。
SceneP1复杂技能和执行阶段未来会快速放大战斗 runtime 复杂度。战斗执行域天然容易爆炸。
ProtocolP2Messages/World 未来可能增胖。契约层一旦混乱会拖累所有 server。
GatewayP2当前健康，主要风险是业务前置化。它应该继续保持无状态接入层。
LoginP2会话模型仍偏简单。这是能力边界问题，不是当前主矛盾。
Router / MgoP2职责清晰，但数据模型未来仍需深化。当前主矛盾不在这里。
ToolingP3生成链复杂但收益明确。短期更适合补文档和可观测性。

P0：现在最值得继续做的

  • World 新业务继续强制走 ValidateRequest → PrepareRuntimeDispatch → DispatchRuntimeCommand。
  • App 层剩余边界桥接继续回收成共享 helper。
  • 不让业务逻辑重新长回流程 owner / callback graph。

P1 / P2：中期应对的风险

  • Scene Combat Runtime 需要继续关注执行阶段边界。
  • Protocol 需要提前控制目录膨胀。
  • Gateway / Login / Router / Mgo 主要是守边界和增强能力。

怎么结合源码索引和 Runtime 页一起看

优先级判断并不是凭感觉拍出来的，而是直接对应到你现在仓库里最关键的接口形态。

如果 DispatchRuntimeCommand 持续稳定，说明 Runtime 下沉方向是对的，World 就还能继续瘦身。
如果请求校验 helper 和 async glue 继续复用，Servers/App 的 P0 价值会持续放大。
如果主链路页、示例页和 Runtime 页开始出现新的 callback graph 味道，就说明设计在倒退。
