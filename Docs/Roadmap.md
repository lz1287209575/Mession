# 后续路线

这份文档不再按“里程碑周报”的方式记录重构历史，只保留当前代码还值得继续推进的方向。

## 1. 收敛异步流程表达

当前 `MFuture / MPromise / MCoroutine` 已经完成基础分层，但业务编排的表达力还不够统一。

后续方向：

- 继续减少 `Then(lambda)` 堆叠
- 让复杂服间流程更多落到显式 `Workflow` / `Coroutine`
- 明确“同步等待”“异步串联”“跨线程调度”的推荐边界

## 2. 强化对象域驱动

当前 Persistence / Replication 已经共用对象域快照能力，但还可以继续推进：

- 让更多对象摆脱 fallback 序列化
- 让属性域标记覆盖更多真实玩法字段
- 统一对象树重建与快照回放路径

## 3. 扩展协议与服务的一致性约束

当前协议目录已经按业务域拆开，后续要继续守住这条边界：

- 新消息进入对应业务域目录
- 协议结构优先 `MSTRUCT + MPROPERTY`
- 避免恢复“大聚合协议头文件”

## 4. 完善验证与可观测性

后续优先补：

- 自动化单元测试
- RPC / 并发 / 对象域专项回归
- 复制与持久化链路日志
- 更明确的错误码、统计与诊断输出

## 5. 继续压缩 `Server` 类职责

虽然当前已经拆出 `Rpc`、`Endpoint` 和 `Domain`，但仍有进一步下沉空间。

方向包括：

- 把复杂业务步骤下沉到工作流对象
- 把跨服协作逻辑收口到服务层
- 保持 `Server` 只承担边界与生命周期职责

## 6. 构建真正可扩展的玩法框架

当前玩家对象树只是第一步。后续需要形成更清晰的玩法扩展模板：

- 新组件如何挂载到对象树
- 新属性如何声明域语义
- 新跨服流程如何接入现有 RPC 运行时
- 新验证脚本如何覆盖新增玩法
